1. Overview

1.1 Purpose

This document describes the architecture of a reusable, AI-driven workflow framework designed to orchestrate complex multi-step business processes. The framework combines traditional Java-based processing with Large Language Model (LLM) powered AI agents using a two-tier architecture:

  • Tier 1 — LangGraph4j: Defines the overall workflow as a directed graph of nodes. Nodes can be AI-powered or non-AI (file download, data transformation, API calls).
  • Tier 2 — Nexus AI (LangChain4j): Within AI nodes, orchestrates multi-agent pipelines with parallel execution, sequential chaining, iterative loops, guardrails, and LLM-driven tools.

1.2 Problem Statement

Modern compliance, KYC, and document processing workflows require increasingly sophisticated capabilities:

  • Workflows involve both infrastructure steps (file downloads, cloud storage, API calls) and AI steps (document analysis, entity extraction, classification).
  • AI steps are themselves complex multi-agent pipelines with parallel execution, iterative refinement, and quality enforcement.
  • Documents must be analyzed, classified, and have information extracted with high accuracy.
  • Workflows must handle large-scale documents that exceed LLM output token limits.
  • Execution must be observable, auditable, and recoverable on failure.
  • Teams must be able to create new workflows without reimplementing orchestration, retry logic, guardrails, or LLM integration patterns.

Without a unified framework, each workflow implementation becomes a standalone system with duplicated infrastructure, inconsistent error handling, and fragmented observability.

1.3 Why Two Tiers?

A single orchestration layer cannot efficiently handle both infrastructure tasks and complex AI agent pipelines. The two-tier approach separates concerns:

Concern Tier 1 — LangGraph4j Tier 2 — Nexus AI
Scope End-to-end workflow: download → process → store → notify Within a single AI node: multi-agent extraction pipeline
Node types Any Java logic — file I/O, API calls, AI nodes, routing decisions LLM agents only — with guardrails, tools, and scope management
State Graph state (shared across all nodes in the workflow) AgenticScope (shared across agents within one AI node)
Error handling Node-level retry, conditional edges, fallback paths Agent-level guardrail retry, reprompt, output rewriting
Parallelism Parallel node execution via graph edges Parallel agent execution via agentFactory.parallel()
Key Principle

Tier 1 answers "what steps does this workflow have?" — Tier 2 answers "how does this AI step work internally?" A workflow node can be as simple as a file download or as complex as a 12-agent extraction pipeline.

2. Architecture Goals

GoalDescription
Reusability Core components (AgentFactory, guardrails, batch tools, LangGraph4j nodes) are shared across all workflows. New workflows compose existing pieces rather than building from scratch.
Extensibility New node types, agent types, guardrails, tools, and workflow patterns can be added without modifying the core framework. The plugin-based architecture supports the open/closed principle.
Modularity Tier 1 nodes are self-contained units. Tier 2 agents are self-contained with declared inputs, outputs, guardrails, and tools. Both can be composed without code changes.
Scalability Automatic output pagination handles documents that exceed LLM token limits. Parallel execution at both tiers maximizes throughput.
Maintainability Prompts are stored in the database (not code). Guardrails are reusable Spring beans. Agent configuration is declarative via AgentSpec. LangGraph4j workflows are defined as graph configurations.
Observability Every node execution and every agent execution within AI nodes is tracked. Execution metadata, inputs, outputs, and timings are persisted to the database. Full audit trail for compliance.

3.2 Tier 1 — LangGraph4j (Workflow Graph)

LangGraph4j provides the top-level orchestration layer. A workflow is defined as a directed graph where:

  • Nodes represent processing steps — each node is a Java class implementing AsyncNodeAction.
  • Edges define the execution order — sequential, conditional, or parallel.
  • Graph State carries data between nodes (file paths, processing results, metadata).

Nodes can be:

Node TypeDescriptionExample
Non-AI Node Standard Java logic — file I/O, API calls, data transformation, routing Document download, GCS upload, result storage, notification
AI Node Delegates to Tier 2 — contains a full Nexus AI agent pipeline CSM extraction (12-agent pipeline), document classification, entity resolution
Decision Node Evaluates a condition and routes to different edges Document size check → small doc path vs. large doc path

3.3 Tier 2 — Nexus AI (Agent Pipeline)

When an AI node executes, it enters Tier 2. Nexus AI (built on LangChain4j) provides:

  • AgentFactory — Creates and composes AI agents from declarative AgentSpec configurations. Supports sequence(), parallel(), and loopAtEnd().
  • AgentSpec — Declares an agent's identity, inputs, output key, tools, guardrails, and LLM parameters.
  • AgenticScope — Shared state container within the AI node. Each agent reads inputs from and writes outputs to the scope.
  • UntypedAgent — Execution interface for both LLM-backed agents and pure Java agents.
  • Guardrails — Input/output validation with retry, reprompt, and rewrite capabilities.
  • Tools — Java objects with @Tool-annotated methods exposed to the LLM (e.g., BatchAccumulatorTool for output pagination).

4. Core Architectural Components

4.1 Tier 1 Components

4.1.1 StateGraph (LangGraph4j)

The StateGraph defines the workflow topology. It registers nodes, edges, and conditional routing logic. Once compiled, it produces an executable graph that manages state flow between nodes.

4.1.2 Workflow Nodes (AsyncNodeAction)

Each node implements AsyncNodeAction<WorkflowState>. Nodes receive the current graph state, perform their processing, and return an updated state. Nodes are Spring-managed beans with full access to dependency injection.

4.1.3 Graph State

A typed state object that carries data between nodes. Each node reads what it needs and writes what it produces. The state is immutable between node boundaries — nodes receive a snapshot and return updates.

4.2 Tier 2 Components

4.2.1 AgentFactory

The central factory for creating and composing agents. Key methods:

MethodDescription
create(AgentSpec) Creates a single UntypedAgent from a spec. Loads prompt from database, configures LLM, registers tools and guardrails.
sequence(outputKey, agents...) Agents execute one after another. Each reads the previous agent's output from the shared AgenticScope.
parallel(outputKey, agents...) Agents execute concurrently. All read from the same scope state; results are written independently.
loopAtEnd(max, predicate, agentA, agentB) Agents A and B execute in a loop until the predicate evaluates to true or max iterations are reached.

These patterns can be nested arbitrarily. A sequence can contain parallel groups, which can contain sub-sequences.

4.2.2 AgentSpec

A declarative configuration object that defines everything needed to create an agent:

PropertyPurpose
name Unique agent identifier. Maps to prompt lookup in database.
inputs(...) Declares which scope keys this agent reads. Framework resolves values from AgenticScope automatically.
outputKey(...) The scope key where this agent writes its result. Downstream agents read from this key.
tools(...) Java objects with @Tool-annotated methods. LangChain4j discovers and exposes them to the LLM.
inputGuardrails(...) Validation functions that run BEFORE the LLM call. Can block, rewrite, or pass through.
outputGuardrails(...) Validation functions that run AFTER the LLM call. Can rewrite output, trigger retry, or reprompt.
maxSequentialToolExecutions Maximum tool calls per LLM turn (critical for batch pagination).
listener(...) AgentMonitor for execution observability — tracks timing, inputs, outputs, errors.

4.2.3 AgenticScope

A shared state container within an AI node's agent pipeline. When agentFactory.sequence() or parallel() is used, the framework creates ONE shared scope. Each agent reads inputs from the scope (matched by its declared input keys) and writes its output to the scope (under its declared output key). This automatic wiring eliminates manual state passing between agents.

4.2.4 Guardrail System

Guardrails implement LangChain4j's InputGuardrail and OutputGuardrail interfaces, providing deterministic validation that does not rely on the LLM following instructions.

Input Guardrail Outcomes:

OutcomeMethodEffect
Passsuccess()Input is valid. Proceed to LLM call.
RewritesuccessWith(text)Input modified (e.g., inject guardrail headers). Proceed with modified input.
Blockfatal(reason)Input invalid. LLM call is blocked entirely.

Output Guardrail Outcomes:

OutcomeMethodEffect
Passsuccess()Output is valid. Pass to next agent.
RewritesuccessWith(output)Output fixed deterministically (e.g., empty string → null). No LLM re-call.
Repromptreprompt(reason, feedback)Output invalid. Re-invoke LLM with feedback appended to the prompt.
Retryretry(reason)Output invalid. Re-invoke LLM with same prompt.
Haltfatal(reason)Unrecoverable error. Halt with exception.

4.2.5 Batch Accumulator Tool

The BatchAccumulatorTool solves a critical scaling problem: when an agent produces output that exceeds the LLM's output token limit (e.g., extracting 200+ records), the entire call fails.

The tool provides LLM-driven output pagination:

  1. BatchResetInputGuardrail runs first — resets the tool's ThreadLocal state for this invocation.
  2. LLM processes the input. If output is small (≤40 records), it returns JSON directly. Tool is never called.
  3. If output is large (>40 records), LLM calls submit_batch(first 40 records), then continues calling for subsequent batches.
  4. LangChain4j's automatic tool execution loop handles each tool call and feeds the result back to the LLM.
  5. After all batches are submitted, LLM returns a text summary.
  6. BatchMergerOutputGuardrail runs last — detects that the tool was used, reads accumulated batches, merges them into a single JSON result, and replaces the summary with the merged data.
Design Decision

The tool is a Spring singleton with ThreadLocal state. One instance is shared across all agents and all requests. Per-invocation isolation is handled by ThreadLocal, and per-agent isolation is handled by the reset guardrail. No wrapper classes are needed.

4.2.6 Custom Java Agents (UntypedAgent)

Not all workflow steps require LLM calls. Pure Java logic (merging, batching, formatting, routing) can be implemented as UntypedAgent and placed directly in sequences alongside LLM agents. The framework treats them identically to LLM-backed agents.

5. Workflow Execution Pattern

5.1 Tier 1 — Workflow Graph Execution

A LangGraph4j workflow executes as follows:

  1. Trigger — External event (API call, file upload, scheduled job) initiates the workflow with initial state.
  2. Graph traversal — LangGraph4j traverses the graph, executing nodes in the order defined by edges.
  3. Node execution — Each node receives the current Graph State, performs its work, and returns updated state.
  4. Conditional routing — Decision nodes evaluate conditions and route to different edges.
  5. Completion — The graph reaches an end node. Final state contains all workflow results.

5.2 Tier 2 — Agent Pipeline Execution (Within an AI Node)

When a Tier 1 AI node executes, it enters Tier 2:

  1. Scope initialization — The AI node reads from Graph State and seeds the AgenticScope with initial inputs.
  2. Agent invocation — For each agent in the pipeline, the framework:
    • Resolves the agent's declared input keys from the AgenticScope.
    • Runs input guardrails (validation, rewriting, blocking).
    • Invokes the LLM (or Java logic for custom agents).
    • If the agent has tools, LangChain4j enters its automatic tool execution loop.
    • Runs output guardrails (validation, retry, rewrite, merge).
    • Writes the agent's output to the scope under its declared outputKey.
  3. Result extraction — The AI node reads the final output from the AgenticScope and writes it back to Graph State.

5.3 Failure and Retry Handling

LayerTierMechanismHandles
Graph Node Retry Tier 1 LangGraph4j conditional edges + retry logic Node-level failures, infrastructure errors, timeout
LLM Provider Tier 2 HTTP-level retry (built into LLM client) 429 rate limits, 500 errors, network timeouts
Output Guardrails Tier 2 reprompt() / retry() Invalid JSON, missing fields, quality violations
Loop Pattern Tier 2 loopAtEnd(max, predicate) Iterative refinement until quality threshold met
Batch Tool Tier 2 submit_batch() tool Output token limit exceeded — LLM paginates automatically

6. AI Agent Integration using Nexus AI

6.1 How Agents Are Created

When AgentFactory.create(AgentSpec) is called:

  1. PromptStore Lookup — The agent's name is used to look up its prompt configuration from the database. This returns the system instruction, user message template, model name, temperature, topP, topK, thinking budget, max output tokens, and response schema.
  2. ChatModel Construction — A ChatLanguageModel is built with the provider-specific factory (e.g., NexusGoogleGenAiChatModelFactory for Gemini). All LLM parameters are configured from the database record.
  3. Agent Builder Assembly — The AgenticServices.agentBuilder() is configured with the chat model, prompts, input keys, output key, tools, guardrails, memory, RAG, and monitoring listeners.
  4. Agent Build — The builder produces a fully configured UntypedAgent ready for invocation.

6.2 How Agents Interact with LLMs

At invocation time, the framework resolves the agent's declared input keys from the AgenticScope, substitutes them into the prompt template, and sends the complete message (system + user) to the configured LLM. The LLM response is captured, run through output guardrails, and written to the scope.

If the agent has tools configured, LangChain4j enters an automatic tool execution loop: the LLM response may contain tool_call directives, which the framework executes and feeds back to the LLM until the LLM produces a final text response.

6.3 How Execution Results Are Stored

  • Scope State — All intermediate and final agent outputs are stored in the AgenticScope, which can be persisted to the database for post-execution analysis.
  • Agent Monitor — The AgentMonitor listener captures execution events (start, complete, error) with timing, input summaries, output summaries, and exception details.
  • Prompt Metadata — The exact prompt version, model parameters, and response are recorded, enabling full reproducibility of any agent execution.
  • Workflow Run Tracking — Each workflow execution is assigned a unique run identifier, enabling end-to-end tracing from trigger to final output.

6.4 Observability, Reproducibility, and Debugging

  • Full Audit Trail — Every agent execution is recorded with its inputs, outputs, model parameters, and timing. Compliance teams can trace how every decision was made.
  • Reproducibility — Given the same prompt version, model parameters, and input, the same output can be reproduced. Temperature=0 ensures deterministic results.
  • Debugging — When an agent produces incorrect output, engineers can examine the exact prompt sent to the LLM, the scope state at that point, and the guardrail evaluation chain.
  • Performance Monitoring — Agent execution times, token counts, and error rates are tracked for capacity planning and optimization.

9. How to Add a New Workflow

Adding a new workflow requires no changes to the core framework. Teams follow these steps:

Step 1: Design the Tier 1 Graph

Identify the high-level steps of your workflow. Determine which steps are AI-powered and which are standard Java logic. Define the graph topology (sequential, conditional branching, parallel).

Step 2: Create Tier 1 Nodes

Implement AsyncNodeAction<YourWorkflowState> for each non-AI node. Register them in the LangGraph4j StateGraph definition.

Step 3: Define Agent Prompts in Database

For each AI agent, insert a prompt record into the PromptStore table with: agent name, system instruction, user message template, model parameters (temperature, topP, topK, thinkingBudget, maxOutputTokens), and response schema.

Step 4: Create AgentSpecs

Step 5: Compose the Tier 2 Pipeline

Use AgentFactory to compose agents into the desired execution pattern within the AI node:

Step 6: Create the AI Node (Bridge)

Create the Tier 1 node that bridges to Tier 2. It reads from Graph State, seeds AgenticScope, invokes the pipeline, and writes results back to Graph State.

Step 7: Wire the Tier 1 Graph

Register all nodes (AI and non-AI) in the LangGraph4j StateGraph, define edges, and compile.

Developer Experience

A typical new workflow with 3-5 agents can be built in 1-2 days: half a day for prompts, half a day for AgentSpecs and Tier 2 composition, and half a day for Tier 1 graph wiring and testing.

10. Benefits of the Architecture

BenefitImpact
Faster Development New workflows are assembled from existing components. Tier 1 nodes and Tier 2 agents are reusable. A 5-agent workflow can be defined in ~30 lines of configuration code.
Clear Separation of Concerns Tier 1 handles infrastructure orchestration (file I/O, storage, APIs). Tier 2 handles AI agent orchestration (LLM calls, guardrails, tools). Each tier can evolve independently.
Standardized Execution Every workflow follows the same pattern: trigger → graph traversal → node execution → AI pipeline → result storage. Consistent behavior across all workflows.
Seamless AI Integration LLM-powered agents and Java logic agents coexist using the same UntypedAgent interface. Teams add AI capabilities without learning new patterns.
Quality Enforcement Guardrails provide deterministic validation independent of the LLM. Input validation, output schema checking, and automatic retry/reprompt are built in.
Automatic Scaling BatchAccumulatorTool handles output token limits transparently. Agents produce correct results whether output is 10 records or 500.
Full Observability Both tiers provide execution tracking. Tier 1 tracks node-level execution. Tier 2 tracks agent-level execution with prompt details. End-to-end traceability for compliance.
Database-Driven Prompts Prompt changes deploy without code changes. Prompt engineers iterate independently from developers. A/B testing of prompt variants is possible.
Reduced Duplication Shared guardrails, tools, orchestration patterns, and Tier 1 nodes eliminate copy-paste across workflows. Bug fixes propagate to all workflows automatically.

11. Future Extensions

11.1 Additional AI Agent Types

  • Multi-modal agents — Native support for vision-based document processing (scanned PDFs, handwritten documents) using multi-modal LLM capabilities.
  • RAG-enhanced agents — Agents augmented with retrieval from vector databases for knowledge-grounded processing.
  • Multi-LLM agents — Agents that fan out to multiple LLM providers and select the best response.

11.2 More Workflow Types

  • Conditional branching — Route execution to different agent paths based on intermediate results.
  • Human-in-the-loop — Pause workflow execution for human review at critical decision points, then resume.
  • Sub-workflow invocation — Workflows that call other workflows as nested steps.
  • Streaming execution — Progressive result delivery as agents complete, rather than waiting for full pipeline completion.

11.3 Improved Orchestration

  • Phase-level checkpointing — Persist scope state after each phase for workflow resume on failure without re-running completed steps.
  • Per-chunk parallel MAP — For extremely large documents, process chunks in parallel with a merge step.
  • Priority-based scheduling — Route high-priority workflows to dedicated agent pools.

11.4 Advanced Monitoring

  • Real-time dashboard — Live visualization of workflow execution, agent timings, and error rates.
  • Cost tracking — Per-agent and per-workflow LLM cost attribution based on token consumption.
  • Quality trends — Historical analysis of guardrail pass/fail rates and critic scores across workflow versions.
  • Alerting — Automated notifications when guardrail failure rates or agent error rates exceed thresholds.
