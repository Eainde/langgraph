1. Overview & Motivation

Prompts are code. They define how LLM agents behave — what they extract, how they classify, what format they return. Yet unlike application code, prompts have no linting, no static analysis, and no automated quality gates.

In practice, prompts silently degrade over time. Someone adds vague language, removes the output schema, or introduces contradictory rules. The LLM starts hallucinating or returning malformed output, and nobody can pinpoint when the regression happened because nobody was testing the prompts themselves.

Prompt Lint brings the discipline of static analysis to LLM prompts:

  • Rule-based, deterministic analysis — no LLM calls, no API keys, no latency
  • 8 orthogonal quality dimensions — from clarity and specificity to injection resistance
  • Weighted scoring by agent type — extraction agents weight groundedness higher; formatters weight output contract higher
  • CI integration — set a quality threshold, run on every commit, fail the build on regressions
  • JUnit assertion API — test prompts the same way you test code

✅ Key Benefits

Analysis runs in milliseconds. No network calls, no API keys, no flaky tests. The same prompt always produces the same score — fully deterministic. The only runtime dependency is Jackson for JSON validation.

2. Quality Dimensions

Prompt Lint evaluates prompts across 8 independent quality dimensions. Each dimension captures a different aspect of prompt quality and runs a series of rule-based checks.

Dimension Code What It Checks
Clarity CLR Role definition, task statement, imperative verbs, vague language detection, output format section, task-before-rules ordering
Specificity SPC Numbered rules, concrete examples, quantified thresholds, boundary conditions, enum constraints
Groundedness GND Source-grounding phrases, anti-hallucination guards, evidence requirements, citation instructions
Output Contract OUT JSON examples present and valid, field-level documentation, schema completeness
Constraint Coverage CON Edge case handling, null/empty handling, error handling, ordering rules, boundary conditions
Consistency CST Template variable alignment with declared inputs, contradictory instructions, terminology consistency
Token Efficiency TKN Prompt length vs complexity ratio, redundancy detection, filler removal
Injection Resistance INJ System/user boundary enforcement, input sanitization, refusal instructions, role-lock phrases

3. Architecture

The library follows a simple pipeline architecture. The PromptQualityAnalyzer orchestrates a set of dimension-specific analyzers, each running independently against the prompt text.

PromptUnderTest → PromptQualityAnalyzer → 8 Dimension Analyzers → PromptQualityReport

Key Components

Component Responsibility
PromptUnderTest Wraps the prompt with metadata: system/user text, declared inputs/outputs, agent profile
PromptQualityAnalyzer Orchestrates all dimension analyzers, aggregates scores using profile weights
AgentTypeProfile Defines dimension weights per agent type (EXTRACTION, CLASSIFICATION, etc.)
PromptQualityReport Holds overall score, per-dimension scores, and all flagged issues
PromptQualityReportRenderer Renders reports as formatted console output (CI-friendly)
PromptQualityAssert Fluent assertion API for JUnit tests

4. Getting Started

Maven Dependency

<dependency>
    <groupId>com.eainde</groupId>
    <artifactId>prompt-lint</artifactId>
    <version>1.0.0</version>
    <scope>test</scope>
</dependency>

Basic Usage

import com.eainde.prompt.quality.PromptQualityAnalyzer;
import com.eainde.prompt.quality.model.AgentTypeProfile;
import com.eainde.prompt.quality.model.PromptUnderTest;
import com.eainde.prompt.quality.report.PromptQualityReport;

import java.util.Set;

// 1. Create the analyzer
var analyzer = PromptQualityAnalyzer.create();

// 2. Wrap your prompt
var prompt = new PromptUnderTest(
    "my-extraction-agent",                  // agent name
    systemPromptText,                       // system prompt
    userPromptText,                         // user prompt
    Set.of("sourceText", "fileNames"),      // declared input variables
    "rawNames",                             // declared output key
    AgentTypeProfile.EXTRACTION             // agent type profile
);

// 3. Analyze
PromptQualityReport report = analyzer.analyze(prompt);

// 4. Check results
System.out.println("Score: " + report.overallScore());
System.out.println("Critical issues: " + report.hasCriticalIssues());

Analyze Multiple Agents

var analyzer = PromptQualityAnalyzer.create();
var renderer = new PromptQualityReportRenderer();

List<PromptUnderTest> prompts = List.of(
    extractorPrompt(),
    classifierPrompt(),
    formatterPrompt()
);

List<PromptQualityReport> reports = analyzer.analyzeAll(prompts);

// Print CI summary
System.out.println(renderer.renderSummary(reports, 0.75));

5. Agent Type Profiles

Each profile adjusts dimension weights to match what matters for that agent type. This means an extraction agent is scored differently than a formatting agent.

Profile Top Weighted Dimensions Use When
EXTRACTION Groundedness (0.25), Specificity (0.15), Output Contract (0.15) Extracting structured data from documents
CLASSIFICATION Specificity (0.20), Constraint Coverage (0.20), Groundedness (0.15) Categorizing inputs into defined classes
FORMATTING Output Contract (0.30), Consistency (0.15), Constraint Coverage (0.15) Transforming data into a specific output format
REVIEW Groundedness (0.20), Specificity (0.15), Output Contract (0.15) Reviewing and validating content
DEFAULT Equal weights (0.125 each) General-purpose agents or when unsure

Custom Profiles

var customProfile = new AgentTypeProfile("MY_AGENT", Map.of(
    "CLARITY",              0.20,
    "SPECIFICITY",          0.20,
    "GROUNDEDNESS",         0.10,
    "OUTPUT_CONTRACT",      0.20,
    "CONSTRAINT_COVERAGE",  0.10,
    "CONSISTENCY",          0.10,
    "TOKEN_EFFICIENCY",     0.05,
    "INJECTION_RESISTANCE", 0.05
));

⚠️ Important

Custom profile weights must sum to 1.0. All 8 dimensions must be included in the weight map.

6. JUnit Integration & CI Usage

Quality Gate Test

import static com.eainde.prompt.quality.PromptQualityAssert.assertThat;

@Test
void promptMeetsQualityBar() {
    var analyzer = PromptQualityAnalyzer.create();
    var report = analyzer.analyze(myPrompt());

    assertThat(report)
        .passesThreshold(0.75)
        .hasNoCriticalIssues()
        .dimensionScoreAbove("GROUNDEDNESS", 0.80)
        .dimensionScoreAbove("OUTPUT_CONTRACT", 0.70)
        .hasFewerIssuesThan(10);
}

Regression Testing

Once you fix an issue, encode it as a test so it doesn't regress:

@Test
void fixedIssueStaysFixed() {
    var report = analyzer.analyze(myPrompt());

    assertThat(report)
        .doesNotHaveIssue("CLR-005")   // output format section was added
        .doesNotHaveIssue("OUT-001");   // JSON example was added
}

Assertion API Reference

Method Description
passesThreshold(double) Asserts overall score ≥ threshold
hasNoCriticalIssues() Asserts no CRITICAL severity issues found
dimensionScoreAbove(dim, score) Asserts a specific dimension score ≥ value
hasFewerIssuesThan(count) Asserts total issue count < value
doesNotHaveIssue(code) Asserts a specific issue code is absent (regression guard)

8. Issue Codes & Severity

Severity Levels

Severity Meaning CI Impact
CRITICAL Prompt is missing something fundamental (e.g., no output format) Should fail the build
WARNING Quality problem that should be fixed (e.g., vague language) Configurable — fail or warn
INFO Improvement opportunity (e.g., add more imperative verbs) Informational only

Issue Code Prefixes

Prefix Dimension Example Issues
CLR- Clarity CLR-004 (vague language), CLR-005 (missing output format), CLR-007 (ambiguous quantifiers)
SPC- Specificity SPC-002 (no quantified thresholds)
GND- Groundedness GND-001 (no source grounding)
OUT- Output Contract OUT-001 (no JSON example), OUT-002 (invalid JSON)
CON- Constraint Coverage CON-001 (no edge case handling)
CST- Consistency CST-001 (template variable mismatch)
TKN- Token Efficiency TKN-001 (redundant instructions)
INJ- Injection Resistance INJ-001 (no system boundary)

9. Use Cases

Use Case Description
CI/CD Quality Gates Set a minimum score threshold and fail the build if any prompt drops below it. Turns prompt quality from a subjective review into an automated check.
Multi-Agent Pipelines Validate all agent prompts in one test class with agent-specific profiles and thresholds. Extraction, classification, and formatting agents each get scored by what matters for them.
Code Review Assistance Include the rendered report in pull requests. Reviewers can immediately see if a prompt change introduced vague language, removed the output schema, or broke variable alignment.
Prompt Engineering Iteration Run the analyzer after each prompt change to see how the score shifts across dimensions. Like a code coverage report — but for your prompts.
Enterprise Governance Teams maintaining shared prompt libraries can enforce minimum quality standards across all agents, with different thresholds per agent type.

10. FAQ

Q: Does this require an LLM API call?

No. All analysis is rule-based and deterministic. There are no LLM calls, no API keys, and no network requests. The library runs entirely in-process.

Q: What Java version is required?

Java 17 or higher. The library uses var, records, and sealed classes from modern Java.

Q: What are the runtime dependencies?

Only Jackson (jackson-databind) for JSON validation in the Output Contract analyzer. No other runtime dependencies.

Q: Can I add custom rules?

Custom agent type profiles (dimension weights) are supported today. Custom rule authoring for individual dimensions is on the roadmap.

Q: How do I integrate this into CI?

Add the dependency with <scope>test</scope>, write JUnit tests using PromptQualityAssert, and set a quality threshold. The tests run as part of your normal mvn test phase. If the score drops below the threshold, the build fails.

Q: How is the overall score calculated?

Each dimension analyzer produces a score between 0.0 and 1.0. The overall score is a weighted average of all dimension scores, where the weights come from the selected agent type profile. For example, the EXTRACTION profile weights Groundedness at 0.25 and Specificity at 0.15.
